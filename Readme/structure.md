# Структура и логика личного кабинета

Документ описывает структуру экрана и поведение интерфейса (мобильная и браузерная версии).

---

## Политика правок: общий код и версии

Личный кабинет — **один код** (один HTML/JS/CSS). Мобильная и браузерная версии — это одна страница, которая по ширине экрана меняет вёрстку (медиа-запросы, показ/скрытие блоков).

| Тип изменения | Пример | Результат |
|---------------|--------|-----------|
| **Общие данные и логика** | Хедер только из БД (без заглушек), логин, счётчики из API | Одна правка → одинаково на мобиле и в браузере. |
| **Только мобильная вёрстка/поведение** | Свайп, футер внизу, скрытие меню, карточки вместо таблицы | Правки в `@media (max-width: 768px)` и в блоках вроде `#dreams-mobile-swipe`. Меняется только на узком экране. |
| **Только браузерная вёрстка** | Таблица мечт, верхнее меню | Стили без медиа или для широкого экрана. Меняется только на десктопе. |

**Как договариваться в задачах:** говорим явно, что правим. «Делаем хедер только из БД» — правка общая (и мобил, и браузер). «Дорабатываем мобильную версию» / «правим свайп» — трогаем только мобильные блоки и стили. «Дорабатываем только десктоп» — только таблица, меню, широкий экран.

---

## Общая трёхсекционная структура (мобильная версия)

Экран личного кабинета на мобильном устройстве делится на **три вертикальные зоны**:

| Секция   | Расположение | Поведение |
|----------|--------------|-----------|
| **Хедер**  | Верх экрана   | Прикреплён к верху (sticky/fixed). Всегда на месте. |
| **Контент**| Между хедером и футером | Занимает всё доступное пространство между ними. Скролл/свайп только внутри контента. |
| **Футер**  | Низ экрана   | Прикреплён к низу (fixed). Всегда на месте. |

Цель: адаптивная вёрстка под любые размеры экрана — хедер и футер статичны, контент заполняет остаток высоты.

---

## Хедер

- Уже сформирован: аватар (свой / бадди в зависимости от режима), имя, статистика (N мечт сбылось, N выполнил), слот бадди / «Я».
- **Кликабельные счётчики:** «N мечт сбылось» — возврат к своим мечтам/шагам; «N выполнил» — открывает витрину мечт (чужие мечты, помочь другим). Оба счётчика кликабельны на мобиле и в браузере.
- На мобильном меню «Мечты / Шаги» скрыто; переключение «Мои мечты» / «Мечты бадди» через слот бадди и тап по аватарке/ссылке «Я».

---

## Политика отображения аватарок и слотов в хедере

**Режимы экрана:** «Мои мечты» (ниже — мои мечты) и «Мечты бадди» (ниже — мечты выбранного бадди). От режима однозначно зависят левый и правый слоты и контент под ними.

| Режим | Левый слот (чей контекст) | Правый слот | Действие по клику |
|-------|---------------------------|-------------|---------------------|
| **Мои мечты** | Моя аватарка + моё имя | Аватарка бадди или кнопка «Добавить бадди» | Правый: переключить на «Мечты бадди» или открыть модалку выбора бадди. Левый: только отображение (загрузка фото временно отключена). |
| **Мечты бадди** | Аватарка бадди + имя бадди | «Я» (ссылка/кнопка) | Правый: переключить на «Мои мечты». Левый: только отображение. |

- **Загрузка аватарки:** временно отключена (клик по аватарке не открывает выбор файла). Позже — только для своего аватара и только когда в левом слоте отображаюсь я (режим «Мои мечты»). Задача «найти место для смены аватарки» зафиксирована в roadmap.
- **Если бадди нет:** правый верхний угол — кнопка «Добавить бадди». По клику — модалка со списком зарегистрированных пользователей (из БД, таблица `users`): аватар (фото или инициалы в кружке), имя и фамилия, кнопка «Пригласить». После отправки приглашения кнопка меняется на «Ждём ответа»; по клику на «Ждём ответа» приглашение отменяется, кнопка снова «Пригласить».
- **Входящий запрос в бадди:** при следующем заходе в кабинет пользователь видит запрос на месте правого слота (потенциальный бадди): кто приглашает, кнопки «Принять» / «Отклонить».
- **Несколько бадди (будущее):** логика рассчитана на переход к нескольким бадди (таблица связей, список бадди в хедере). Сейчас в UI — один слот справа и один выбранный бадди; смена схемы БД и списка бадди — отдельные шаги в roadmap.

### Реализация приглашения в бадди (теория)

**БД:** Таблица запросов, например `buddy_requests`: `id`, `from_user_id`, `to_user_id`, `status` (`pending` | `accepted` | `declined`), `created_at`. Пока один бадди — при принятии запроса обновляем `users.buddy_id` у обоих (симметричная связь). Для нескольких бадди позже — таблица связей `user_buddies` и перенос логики принятия туда.

**API:** (1) `GET /users` или `GET /users/list` — список пользователей для модалки (из таблицы `users`, без текущего и без уже бадди), поля: id, name, surname, avatar_path (или инициалы). (2) `POST /buddy_requests` с телом `{ to_user_id }` — отправить приглашение (from = текущий пользователь). (3) `GET /buddy_requests` — мои запросы: входящие со статусом `pending` (показать в правом слоте при заходе в кабинет), исходящие (для кнопки «Ждём ответа» и отмены). (4) `PATCH /buddy_requests/:id` с телом `{ status: "accepted" | "declined" }` — принять или отклонить. При `accepted` — создаём связь (обновляем `buddy_id` у обоих пользователей или пишем в `user_buddies`). (5) Отмена исходящего: `DELETE /buddy_requests/:id` или `PATCH ... { status: "cancelled" }` — по клику «Ждём ответа» возвращаем кнопку в «Пригласить».

**Фронт:** Модалка «Добавить бадди»: при открытии запрос `GET /users/list`, рендер списка (аватар или круг с инициалами, имя фамилия, кнопка «Пригласить»). По клику «Пригласить» — `POST /buddy_requests`, кнопка меняется на «Ждём ответа». По клику «Ждём ответа» — отмена запроса (DELETE/PATCH), кнопка снова «Пригласить». При заходе в кабинет проверяем `GET /buddy_requests`; если есть входящие `pending`, в правом слоте показываем блок «Запрос от [имя]» с кнопками «Принять» / «Отклонить»; по нажатию — `PATCH` с нужным статусом и обновление хедера (появление бадди в слоте при принятии).

## Контент

- В основном отображает **мечты**: одна мечта = одна карточка.
- **Свайп влево/вправо**: переключение между карточками мечт. Одна мечта занимает **всё доступное поле контента** — соседние карточки не видны (нет «кусков» соседнего блока по краям).
- На десктопе контент — таблица мечт или те же карточки по логике выбранного блока (Мои / Бадди / Расписание).

---

## Цепочка: мечта → шаги → расписание

**Общая схема:** Пользователь создаёт мечту → разбивает её на шаги → шаги попадают в ежедневное расписание (раздел «Шаги» / Расписание). Открыл расписание, увидел, что сделать сегодня, отметил выполнение.

**Два типа декомпозиции:**
- **Простые мечты:** шаги добавляются по одному (название, дедлайн, галочка «выполнил»). Кнопка «Добавить шаг» открывает форму одного шага.
- **Особые мечты:** требуют своего интерфейса и логики (например, финансы — таблица по месяцам с планом/фактом; книги — список книг, статусы, виртуальные строки в расписании). Для таких целей в справочнике **`steps_rules`** хранится правило (rule_code, параметры). В **Readme/rules.md** — полная текстовая инструкция для людей и агентов (как создавать шаги, что показывать в мечте и в расписании). Когда пользователь нажимает «Добавить шаг» у особой мечты, система определяет тип мечты по полю **`dreams.rule_code`**, находит инструкцию в `steps_rules` и реализует декомпозицию в соответствии с ней (другой пользователь или новый админ, открыв инструкцию, может выполнить ту же задачу без искажений).

Идентификация мечты как «особой» (привязка к rule_code) и момент, когда система подставляет нужный интерфейс при «Добавить шаг», уточняются в roadmap.

## Шаги в Мечтах и Расписание

**Источник данных:** Обычные шаги хранятся в таблице `dreams_steps` и привязаны к мечте (`dream_id`). Для мечт с особым rule_code (например, книги) часть «шагов» может генерироваться виртуально (см. GET /schedule) и храниться в своих таблицах (dream_books, dream_books_log). Они отображаются в двух местах:

| Место | Назначение |
|-------|------------|
| **Шаги в «теле» мечты** (аккордеон в таблице/карточке мечты) | Показать шаги, относящиеся к **конкретной мечте**: название шага, дедлайн, галочка, редактирование. Для финцели — таблица ВЗНОС/СУММА/Итог с вводом факта. |
| **Расписание** (отдельный раздел, в меню — «Расписание», ранее «Шаги») | **Сводка всех шагов по всем мечтам** пользователя = дорожная карта. Открыл, посмотрел, что сделать сегодня, отметил выполнение и пошёл дальше. |

**Правило:** Для обычных мечт шаги в «теле» мечты и в Расписании — одни и те же записи (`dreams_steps`). Расписание — объединённый вид по датам и мечтам. Для особых типов (финансы, книги) в «теле» мечты показывается свой интерфейс (таблица, список книг), а в Расписании — понятные строки (например, «Внеси 51 т.р.» или «Читать «Книга Ксении» 15 мин»), в т.ч. виртуальные (собираются на бэкенде по правилу).

### Особенности для шагов финцели в Расписании

В «теле» мечты финцель отображается **таблицей** (месяц, план, факт, итог). В Расписании — **понятная одна строка** на месяц: «Внеси 51 т.р.»: открыл Расписание 31 марта и сразу ясно, что делать.

1. **Запись в Расписании:** одна строка на месяц финцели, текст **«Внеси N т.р.»** (N — актуальная сумма к внесению: накопительный план минус накопительный факт на конец того месяца). Дедлайн шага — **последний день месяца** (31.03 для марта). Шаг **появляется в списке в день дедлайна** (для марта — 31 марта); все шаги в Расписании видны скроллом (можно посмотреть до конца года).
2. **Порядок и фильтр:** в начале списка — **актуальные, сегодняшние** шаги. Отдельный **фильтр «Просроченные»** — шаги с дедлайном в прошлом, не выполненные.
3. **Поведение при отметке:** только **галочка**. По нажатию — **модалка**: «Я правильно понимаю, что ты вносишь сумму ровно N руб?» (N — актуальное значение). Варианты: **«Да»** или **«Указать другую сумму»** (поле ввода). При «Да» — сохраняем `fact_amount = N`, шаг выполнен. При другой сумме — сохраняем введённую сумму, шаг выполнен. После сохранения открывается следующий месяц с новой актуальной суммой и датой = последний день следующего месяца.
4. **Связь с таблицей в Мечте:** если пользователь ввёл факт в таблице мечты (ячейка «ВЗНОС Факт»), соответствующий шаг в Расписании считается выполненным (галочка ставится автоматически), следующий месяц показывается как «Внеси X т.р.» с датой = последний день месяца.

## Футер (мобильная версия)

Одна строка, три элемента в ряд:

| Элемент | Описание |
|---------|----------|
| **Счётчик мечт** | Формат `2 / 11` (текущая карточка / всего). Слева. |
| **Кнопка «+»** | Зелёная кнопка с плюсиком — «Добавить мечту». Правее счётчика, в одной строке. |
| **Выйти** | Малозаметная кнопка выхода. Справа внизу (в той же строке футера). |

Футер показывается на мобильной версии; на десктопе кнопка «Выйти» может оставаться в другом месте (например, под контентом или в меню).

## Логика переключения контента

- **Мои мечты** — список мечт текущего пользователя; счётчик и свайп по ним; кнопка «+» открывает форму добавления мечты.
- **Мечты бадди** — список мечт бадди; счётчик и свайп по ним; кнопка «+» не добавляет мечту (можно скрыть или отключить).
- **Расписание** — сводный список шагов по всем мечтам (текущие/все/выполненные); футер может показывать только «Выйти» или упрощённый вид. Ранее раздел назывался «Шаги».
- **Витрина мечт** — открывается по клику на «N выполнил» в хедере. Чужие публичные мечты, фильтры: Новые, Помогаю, Все, Избранные, Просмотренные. Карточки с кнопкой «Хочу помочь»; модалка с готовым сообщением и кнопками мессенджеров (Telegram, WhatsApp, VK — если автор указал при регистрации). Просмотры, избранное и намерение помочь хранятся в БД (user_dream_views, user_dream_favorites, user_dream_help_intent). Факт — «реально помог» — в dreams_log. Клик по «N мечт сбылось» возвращает к своим мечтам. Публичная витрина (dreams.html) — без авторизации; захотел помочь → регистрация в ЛК.
