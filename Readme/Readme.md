### Разделение агентов: локалка и прод

Чтобы не путать задачи и ответственность:

*   **Forge (локальный агент):** работает на ноутбуке Макса в песочнице `~/Apps/island`, делает разработку, правки кода, документацию, локальные проверки, подготовку коммитов.
*   **Продагент (серверный агент):** работает на прод-сервере в директории проекта на сервере, принимает изменения из Git (`git pull`), выполняет продовые действия (перезапуск сервиса, проверка состояния, при необходимости миграции по регламенту).

**Правило идентификации для серверного агента:** если ты читаешь этот `Readme` на проде — ты и есть **Продагент**. Действуй в рамках продовой роли, не смешивай её с задачами локальной разработки Forge.

#### Проверка роли по окружению (обязательно перед началом задачи)

**Роли взаимоисключающие. Агент не может быть одновременно Forge и Продагентом.**

1. Если текущий сервер/хост соответствует прод-окружению с IP **`188.225.44.48`** — роль агента: **Продагент**.
2. Если работа идёт в локальной песочнице Макса (проект `~/Apps/island`) и это не прод-хост `188.225.44.48` — роль агента: **Forge**.
3. Если есть сомнение:
   - проверить IP/хост машины;
   - посмотреть путь проекта и контекст терминала;
   - только после этого выбирать роль и выполнять действия.

**Приоритет правила:** проверка IP `188.225.44.48` имеет приоритет над названием агента/сессии. На этом IP агент всегда работает как **Продагент**.

**Жёсткая фиксация:**
*   На локальном компьютере Макса агент = **Forge** (и только Forge).
*   На прод-сервере `188.225.44.48` агент = **Продагент** (и только Продагент).
*   Формулировка «если по-простому зови меня Форж» на проде не применяется.
*   **Продагент не наводит “порядки” в коде на сервере** (рефакторинг, перенос файлов, переименование структуры, массовые правки README/архитектуры).
*   Любые структурные изменения делает только **Forge в песочнице**; затем изменения проходят через GitHub (`commit/push`) и только после этого подтягиваются на прод (`git pull`).
*   Продагент выполняет только продовый контур: `pull`, перезапуск сервисов, проверка состояния, применение заранее подготовленных изменений по регламенту.

#### SOP: Forge ↔ Продагент (рабочий цикл)

1. **Постановка задачи:** Макс помечает задачу как `ЛОКАЛКА/FORGE` или `ПРОД/PRODAGENT`.
2. **Проверка роли:** перед началом агент сверяет окружение (локалка vs IP `188.225.44.48`) и подтверждает единственную роль.
3. **Разработка:** любые изменения кода/структуры/README делает только Forge в `~/Apps/island`.
4. **Локальная проверка:** Forge проверяет работоспособность, обновляет документацию и счётчик версии (`.app-version`) по правилам проекта.
5. **Публикация:** Forge делает `git add` -> `git commit` -> `git push origin main`.
6. **Обновление прода:** Продагент выполняет `git pull origin main` и перезапуск нужного сервиса по регламенту. **Если приложение на проде переведено на Docker (решение Кроноса):** из каталога проекта с актуальным **`.env`** (боевые `DB_*` на внешнюю БД, без `docker-compose.dev.yml`):  
   `docker compose up -d --build`  
   Поднимается только сервис **`app`** из **`docker-compose.yml`**; локальный Postgres в compose **не** используется.
7. **Проверка прода:** Продагент проверяет статус сервиса, доступность страниц/API и сообщает Максу результат.
8. **Эскалация:** если на проде нужна правка кода, Продагент не правит вручную, а возвращает задачу Forge в песочницу.

### Окружение и запуск

Чтобы любой (в т.ч. AI) мог знать всё о сервере и проекте без лишних вопросов:

*   **Путь к проекту на сервере:** `/home/makc/Apps/island`
*   **Запуск приложения:** служба `fastapi.service` стартует так (рабочая директория — корень проекта):
    ```bash
    /usr/bin/python3 -m uvicorn main:app --host 0.0.0.0 --port 8000
    ```
    Команды перезапуска и логов — ниже в блоке «Пульт управления». При желании их можно заранее проверить вручную.
*   **Переменные окружения (.env):** в корне проекта файл `.env` (не коммитить). Имена переменных: `DB_HOST`, `DB_USER`, `DB_PASS`, `DB_NAME` — их читает `main.py` для подключения к PostgreSQL. Опционально: `DB_SSLMODE` (например `require` или `prefer`). При ошибке «SSL connection has been closed unexpectedly» попробуй `DB_SSLMODE=require`; при `verify-full` проблемы с сертификатом могут вызывать разрывы.
*   **Ограничения и производительность:** приложение и БД на разных хостах (сервер приложения и сервер БД — см. выше). Пула соединений к БД нет: каждый запрос к API создаёт новое подключение к PostgreSQL. При росте нагрузки имеет смысл добавить пул соединений.
*   **Unit-файлы (файлы служб systemd):** лежат в `/etc/systemd/system/` — например `fastapi.service`, при необходимости `frontend.service`. Чтобы посмотреть путь к конкретной службе: `systemctl show fastapi.service -p FragmentPath`.

### Резервное копирование БД

*   **Папка:** `db_backups/` в корне проекта.
*   **Скрипт:** `db_backups/bu_db.sh` — по расписанию (cron) сравнивает текущий размер БД с последним сохранённым; если размер изменился, создаёт дамп `default_db_YYYYMMDD_HHMMSS.dump` (формат `pg_dump -F c`).
*   **Лог:** `db_backups/backup.log` — дата и имя созданного файла или «размер не изменился, бэкап пропущен».
*   **Восстановление / перенос:** последний файл `*.dump` в `db_backups/` можно использовать для восстановления на этом сервере или для переноса на другой (например, при миграции на Timeweb): `pg_restore -h ... -U ... -d ... -F c /path/to/latest.dump`. Учитывать, что дамп может быть на несколько часов отставать от текущего состояния БД (если cron раз в сутки).

## Хедер личного кабинета (источник данных)

Все значения в хедере берутся только из БД (без заглушек).

| Элемент | Источник | Описание |
|--------|----------|----------|
| **Инициалы / Имя** (например: СЩ, Света Щербинина) | **users** | `name`, `surname`; инициалы — первые буквы имени и фамилии. |
| **N мечт всего** | **dreams** | Количество записей `dreams` с `user_id` текущего пользователя. |
| **Сбылось: N мечт \| M раз** | **dreams_log** | По мечтам пользователя: число уникальных мечт с хотя бы одной записью в `dreams_log` и общее число записей («раз»). |
| **1 бадди** / **0 бадди** | **users** | Наличие бадди: у текущего пользователя заполнено поле `buddy_id` (1 бадди) или нет (0 бадди). |
| **исполнил / N / мечту, мечты, мечт** (три строки) | **dreams_log** | Число записей в `dreams_log`, где `fulfilled_by_user_id` = текущий пользователь и мечта принадлежит другому. Строки: «исполнил», число N, склонение: 1 мечту, 2–4 мечты, 0/5+ мечт (правила в `.cursor/rules/header-helped-declension.mdc`). |

Данные для мечт и счётчиков «сбылось» / «чужих исполнил» приходят в ответе API `GET /dreams?user_id=...`: поля `dreams`, `dreams_fulfilled_count`, `dreams_fulfilled_times`, `dreams_fulfilled_by_me`. Бадди и имя — из ответа `POST /login` (текущий пользователь).

   * Добавлен жёсткий CSS-guard для `edit-series`: принудительно скрываются `#modal-add-step-single-block` и `#modal-add-step-mid-column`, а четыре нужных блока (`from/to/time-from/time-to`) принудительно показываются, чтобы не появлялся «пустой шестой» слот.

94. **Серийное редактирование: считаем только «текущий и дальше», без реакций (13.04.2026)**
   * Для кнопки **«N шагов»**, счётчика и сохранения серии учитываются только шаги серии, которые: **не удалены**, **не отмечены выполненными** и идут **от даты текущего открытого шага и дальше**.

95. **Edit-series: поля дат и времени выровнены по одной горизонтали (13.04.2026)**
   * В верхнем ряду `Начало | Конец | Время с | Время по` слоты ввода унифицированы: одинаковая высота контейнера и инпута (`38px`), одинаковое вертикальное выравнивание и отступы. Поля времени больше не «проседают» ниже полей даты.

96. **Модалка шага: убрана служебная подсказка про календарь (13.04.2026)**

## 🔒 Правила разработки (Rules)
1.  **Безопасность:** Файл `.env` никогда не попадает в Git. Пароли в базе не храним открытым текстом (в будущем подключим хеширование).
2.  **Git Flow:** Изменения сначала тестируются локально или на сервере, затем фиксируются (`git commit`) и отправляются в репозиторий (`git push`).
3.  **Code Style:** Чистый код, понятные имена переменных, обработка ошибок через `try/except`.
4.  **Документация БД:** При любом изменении схемы или данных в БД (миграции, новые таблицы/колонки/справочные значения) сразу вносится соответствующая запись в [Readme/tables.md](tables.md).
5.  **Обновление Readme:** После значимых изменений (фичи, баги, UI, БД) — сам обновлять Readme/Readme.md (раздел «Что сделано»), tables.md (при изменении схемы). Roadmap — в БД (таблица `roadmap`), изменения через roadmap.html или API. Правило: [.cursor/rules/update-readme.mdc](.cursor/rules/update-readme.mdc).
6.  **Телефоны в БД:** Единый формат — без пробелов, слитно: `+79998880011`. Нормализация: скрипт `scripts/normalize_phones_and_passwords.py` (приводит к +7…, при необходимости задаёт пароль «последние 4 цифры» пользователям без пароля).
7.  **Миграции и папка _sql:** Выполненная миграция (через `run_migrate.py`) самоликвидируется — файл удаляется из `_sql/`. Проверить, делал ли пользователь миграцию, можно по наличию файла в `_sql/`: нет файла — миграция уже выполнена; файл есть — нужно выполнить.
    *   **Агент (Forge) умеет выполнять миграции сам.** После добавления или правки SQL-файла в `_sql/` агент должен сам запустить `python3 run_migrate.py _sql/имя_файла.sql` (из корня проекта, где есть `.env`). Не перекладывать эту операцию на пользователя: одна команда в терминале — часть задачи «внести изменения в БД».
    *   Если пользователь пишет «перезапустил, результата не вижу» и речь про изменения из миграции — напомнить: сначала выполнить миграцию (или выполнить её самому, если файл ещё в `_sql/`), затем перезапуск сервиса.
    *   **405 Method Not Allowed** при создании шагов финцели (кнопка «Создать шаги» в модалке): обычно значит, что на сервере (23.172.217.180) запущена старая версия backend без маршрута `POST /dreams/{id}/steps/finance`. Нужно задеплоить актуальный код и перезапустить службу: `sudo systemctl restart fastapi.service`.
8.  **Стратегия важнее тактики:** Если в процессе выполнения текущей задачи выявлена системная проблема (повторяющаяся, архитектурная, инфраструктурная, прав доступа), агент должен приостановить тактическое выполнение и предложить системное решение. Если упирается в границы (безопасность, права, ограничения среды), агент обязан явно уведомить Макса, что системное решение сейчас недоступно, и указать точную причину/ограничение.
9.  **Папка `errors/`:** После фикса ошибки всегда предложить удалить соответствующий скрин/артефакт из `errors/` для чистоты директории; удалять только после подтверждения пользователя.
10. **Деплой на прод:** Из песка в прод уходит только `prod-safe` код через Git (`git pull --ff-only`). Данные БД не синхронизируются через Git; обновление песка данными прода — только отдельным дампом и отдельной процедурой с бэкапом.
11. **Уточняющие вопросы обязательны:** Если задача сформулирована неясно, есть спорный момент или возможны разные трактовки — сначала задать уточняющие вопросы, затем выполнять изменения.
12. **Модалка шага (`#modal-add-step`, `index.html`): «Повторять» vs серия.** Источник правды по UX — этот пункт, а не отдельные файлы в `.cursor/rules`.
    * На оверлее задаётся `data-step-modal-ui`: **`create`** | **`edit-single`** | **`edit-series`** (см. `openAddStepModal` / `openEditStepModal`).
    * **Одиночный шаг:** режимы `create` или `edit-single` (шаг не входит в серию из нескольких экземпляров). Показывается кнопка **«Повторять»** — блок периода и создание серии шагов (или развёртывание одиночного в серию — по текущей логике сохранения).
    * **Серийные шаги:** режим **`edit-series`** — у шага больше одного члена серии в данных (`countStepSeriesMembers`, `series_id` и т.д.). Кнопка **«Повторять» не показывается** (`#modal-add-step-repeat-only-wrap` скрыт; стили по `[data-step-modal-ui="edit-series"]`). Вместо неё — две кнопки в том же стиле: **«1 шаг»** (текущий) и **«N шагов»** (вся серия); **«1 шаг»** — один `PATCH` (`scope: single`); **«N шагов»** — по одному `PATCH` на каждый член серии (см. п. 68 в «Что сделано»).
    * В **`edit-series`** верхнее поле даты — подпись **«Шаг за»** (дедлайн открытого экземпляра); «Дата окончания» во второй строке — конец серии (для пересчёта N), **не** то же поле. Кнопка **«N шагов»** при сохранении: фронт шлёт **отдельный** `PATCH` на каждый шаг серии с **своим** `deadline` из периода (и общими названием/временем), число дат в периоде должно совпасть с числом шагов. Массовый `PATCH` с `scope: all_series` по-прежнему **не** применяет одну дату ко всем (см. п. 67 в «Что сделано»).
    * **Нельзя** одновременно показывать «Повторять» и пару «1 шаг / N шагов». Для `edit-series` не возвращать «Повторять» без отдельного решения в ТЗ.
13. **Папка `_trash` (корзина):** устаревшие или выведенные из оборота файлы переносим в `_trash/`, а не держим в корне рядом с рабочим кодом. **Удаление из корзины:** не реже одного раза в месяц просматривать `_trash/`; всё, что **пролежало в корзине более 30 календарных дней** без необходимости вернуть в проект, **удалять окончательно** (из репозитория и с диска), если Макс явно не попросил хранить дольше. Ориентир по давности — дата последнего изменения файла (`mtime`) или дата переноса в `_trash` по истории Git. Корзина не заменяет VCS: важное сохраняем коммитами до переноса в `_trash`.
14. **Прод-миграции БД только additive (без потери данных):** На проде миграции должны расширять/дополнять схему и данные, а не затирать существующее. Любые потенциально разрушительные операции (`DROP`, массовые `DELETE/UPDATE`, восстановление дампа поверх рабочей БД) запрещены без отдельного явного согласования. Базовый порядок: свежий бэкап -> миграция -> проверка.

**Опыт: почему агент раньше не запускал миграции сам.** В инструкциях было сформулировано так: «напомнить пользователю выполнить миграцию» / «файл есть — нужно выполнить» — без явного «агент может и должен выполнить сам». Агент следовал букве: предлагал команду и ждал действий пользователя. После того как пользователь один раз попросил «дай команду на миграцию» и агент её выполнил, стало ясно: (1) у агента есть доступ к терминалу и к `.env` в корне проекта, `run_migrate.py` успешно отрабатывает; (2) выполнение миграции — не «особо опасная» операция, требующая только ручного подтверждения, а часть обычного цикла «написал SQL → применил к БД»; (3) явная фиксация в Readme («агент умеет и должен выполнять миграции сам») убирает неопределённость и в следующий раз агент будет запускать миграцию по умолчанию, не перекладывая на пользователя. **Вывод на будущее:** если в проекте есть повторяемая операция (миграция, скрипт, сборка), и у агента есть все данные и инструменты для её выполнения — явно записать в правила/Readme, что агент делает её сам в рамках задачи, если пользователь не попросил иначе.

## 🖥 Локальный запуск (разработка)

Не коммить `.env` в Git.

### Docker (рекомендуемый контур для песочницы)

Файлы:

| Файл | Назначение |
|------|------------|
| `Dockerfile` | Единый образ: зависимости, код, `uvicorn main:app --host 0.0.0.0 --port 8000`. |
| `docker-compose.yml` | **Только `app`:** порт `127.0.0.1:8000:8000`, переменные из `.env`. Без сервиса БД — подходит для прода с внешней PostgreSQL. |
| `docker-compose.dev.yml` | **Только песочница:** добавляет локальный **Postgres** и монтирование кода `.:/app` для разработки. На прод **не** использовать. |

**Песочница с локальной БД в Docker** (в `.env` задай `DB_PASS`; `DB_USER` / `DB_NAME` по желанию, по умолчанию совпадают с контейнером `db` — см. `docker-compose.dev.yml`):

```bash
cd ~/Apps/island
docker compose -f docker-compose.yml -f docker-compose.dev.yml up --build
```

Первичная схема БД: как раньше (`init.sql` / дамп / `run_migrate.py` из контейнера: `docker compose -f docker-compose.yml -f docker-compose.dev.yml exec app python3 run_migrate.py ...` при наличии файлов в `_sql/`).

**Песочница: только app в Docker, БД снаружи** (в `.env` укажи `DB_HOST` и остальные `DB_*` на доступный Postgres):

```bash
docker compose up --build
```

### Продагент (Docker на сервере, без локального Postgres в compose)

В каталоге проекта на проде — актуальный `.env` с **внешним** `DB_HOST` (и без второго compose-файла с `db`):

```bash
git pull --ff-only origin main
docker compose up -d --build
```

Убедиться, что в команде **нет** `-f docker-compose.dev.yml`.

### Без Docker

1. **Backend:** в корне проекта создай `.env` с переменными (хост БД, пароль и т.д.), затем:
   ```bash

   ```
2. **Сайт:** в браузере открывай `http://localhost:8000/` (корень = кабинет) или `http://localhost:8000/dreams.html` (витрина), `http://localhost:8000/index.html` (кабинет явно).
