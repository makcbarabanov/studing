# RFC: Редактирование повторяющихся шагов

Дата: 10.04.2026  
Статус: draft (на согласовании)  
Цель: ввести предсказуемое редактирование одиночных и повторяющихся шагов без двусмысленности.

## 1) Проблема

Сейчас шаг редактируется как одиночная сущность. Для повторяющихся шагов не хватает явного выбора области применения:

- изменить только текущий шаг;
- изменить текущий и все следующие;
- изменить всю серию.

Из-за этого пользователь не понимает, насколько массовым будет изменение.

## 2) Принцип

Редактирование шага всегда выполняется с явным `scope` (областью применения).  
По умолчанию: безопасный вариант `single` (только текущий шаг).

## 3) Модель данных

Добавить поля в `dreams_steps`:

- `series_id` (UUID/TEXT, nullable) — идентификатор серии повторяющихся шагов.
- `series_index` (INT, nullable) — номер шага в серии (1..N).
- `series_total` (INT, nullable) — общий размер серии на момент создания.

Для одиночного шага:

- `series_id = NULL`
- `series_index = NULL`
- `series_total = NULL`

Для серийных шагов:

- все шаги одной серии имеют одинаковый `series_id`.

## 4) API-контракт (предложение)

### PATCH `/dreams/{dream_id}/steps/{step_id}`

Тело:

```json
{
  "title": "Отбой до 23:00",
  "deadline": "2026-04-10",
  "start_time": "08:15",
  "end_time": "08:30",
  "scope": "single"
}
```

`scope`:

- `single` — только `step_id`
- `from_current` — шаги той же серии с `series_index >= current`
- `all_series` — все шаги той же серии

Для шага без серии:

- `scope` принудительно трактуется как `single`.

Ответ:

```json
{
  "ok": true,
  "updated": 7,
  "scope": "from_current"
}
```

## 5) UI/UX (модалка редактирования)

В режиме редактирования:

- если шаг одиночный: кнопка `Повторять` доступна (можно сделать серию-копию);
- если шаг серийный: показывать блок `Применить изменения`:
  - `Только этот шаг`
  - `Этот и следующие (N)`
  - `Всю серию (M)`

Где:

- `N` = количество шагов в серии от текущего до конца;
- `M` = общее количество шагов в серии.

По умолчанию выбирать `Только этот шаг`.

## 6) Правила массового применения

При `from_current` / `all_series`:

- `title`, `start_time`, `end_time` — применяются массово.
- `deadline`:
  - для `single` редактируется напрямую;
  - для массового режима сейчас не меняется (чтобы не ломать сетку серии), пока не внедрён отдельный механизм сдвига серии.

Это защищает от непредсказуемых календарных сдвигов.

## 7) Edge cases

- Если в серии есть удалённые шаги (`deleted=true`) — не включать их в массовое обновление.
- Если пользователь изменяет только время — не трогать `deadline`.
- Если `start_time > end_time` — ошибка 400.
- Если серия распалась (часть шагов без `series_id`) — работать только по фактически найденным шагам.

## 8) План внедрения

1. Миграция БД (`series_id`, `series_index`, `series_total`).
2. Обновление создания повторяющихся шагов: присваивать `series_id` и индексы.
3. Расширение PATCH с `scope`.
4. UI: блок выбора области применения в модалке редактирования.
5. Тест-кейсы:
   - одиночный шаг,
   - серия из 1 шага,
   - серия из N шагов,
   - `single/from_current/all_series`.

## 9) Критерий готовности

- Пользователь в каждом редактировании явно понимает область применения.
- Нет «тихих» массовых изменений.
- Поведение единообразно в desktop и mobile.
