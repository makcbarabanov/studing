# UI-стандарты Острова

**Назначение:** единый источник правды по повторяющимся визуальным паттернам в ЛК и связанных страницах. При новом меню, переключателе или **поле даты** — сначала сверка с этим файлом, затем переиспользование тех же классов/токенов, а не копирование «на глаз».

**Где в коде:** основной фронт кабинета — `index.html` (встроенные `<style>` и скрипты). CSS-переменные бренда объявлены в `:root` в начале `<style>`.

---

## 0. Фирменные цвета (токены)

Семантика «острова»: **зелёный акцент** (рост, действие, подтверждение), **сине-бирюзовая** палитра календаря **«Остров»**, **серый/нейтральный** для вторичного текста, **красный** — опасное действие.

| Токен CSS | Пример/назначение |
|-----------|------------------|
| `--island-accent` | `#4CAF50` — основной зелёный: кнопка по умолчанию, прогресс, активные фильтры, акценты. |
| `--island-accent-hover` / `--island-accent-dark` / `--island-accent-deep` | Темнее зелёного: hover, обводка, заголовки `h3` в модалках, текст на контурах. |
| `--island-accent-soft` | Светлый мятный фон для hover у контурных кнопок, подсветка выбора. |
| `--island-text` / `--island-text-muted` | Основной и приглушённый текст. |
| `--island-surface` | Фон карточек/модалок — белый. |
| `--island-secondary-bg` / `--island-secondary-border` / `--island-secondary-text` | Вторичные кнопки («Отмена», «Пропустить»): мягкий серо-зеленоватый фон, нейтральная рамка. **Не** использовать плоский `#888` для фона кнопок в ЛК. |
| `--island-link` | `#2196F3` — ссылки, как в глобальных стилях. |
| `--island-danger` / `--island-danger-hover` | Красные кнопки/опасные действия (класс `.btn-danger` в ряду `.modal-actions`). |
| Календарь §3 | `--island-cal-accent` и существующие в коде `#01579b`, `#81d4fa`, градиент попапа — только для **поля даты «Остров»** и сетки календаря. |

Фон страницы кабинета (`body`: `#e8e4df` + фоновое изображение) и контейнера (полупрозрачный белый) заданы в `index.html`; при выносе стилей в отдельный файл — сохранить контраст читаемости.

## 0.1. Кнопки в модалках (`.modal-actions`)

Единая система в `index.html` — все кнопки внизу модалок (`.modal .modal-actions`) оформляются только через эти **классы**, без инлайн-`background` для цвета кнопок.

| Роль | Классы / правило | Внешний вид |
|------|------------------|------------|
| **Основное действие** (Сохранить, Подтвердить, и т.д.) | Без доп. класса; не `.secondary` и не `.island-btn-outline` и не `.btn-danger` | Заливка `var(--island-accent)`, белый текст, лёгкая тень, hover темнее. |
| **Второстепенно** (Отмена, Пропустить, «Нет») | `class="secondary"` | Светлый фон `--island-secondary-bg`, рамка `--island-secondary-border`, текст `--island-secondary-text` — не «мёртвый» серый. |
| **Контур / выбор варианта** (в т.ч. **Все / Прошлые / Будущие** в `#modal-step-series-delete`) | `class="island-btn-outline"` | Белый/поверхностный фон, зелёная обводка и типографика; hover — мягкий `--island-accent-soft`. |
| **Опасно** (редко) | `class="btn-danger"` | Красная заливка. |

- Ряд кнопок: контейнер `.modal-actions` (по умолчанию `justify-content: flex-end`). Для группы вариантов слева — **`.modal-actions.modal-actions--choice`**.  
- `disabled` у контурных кнопок: приглушенный стиль, без зелёного hover.  
- **Глобальное** правило `button` на странице — зелёный; в модалках уточняется селектором `.modal-actions button` (и классами выше), чтобы «второстепенные» и «контур» не унаследовали только зелёный.  
- Другие HTML в репозитории (`roadmap.html`, `dreams.html`, …): при правках в том же визуальном контуре — ориентироваться на эти токены и смыслы, по возможности дублируя `:root` или подключая общий стиль, когда вынесем CSS.

## 1. Сегменты-фильтры (активный = зелёный фон)

**Эталон:** блок фильтра мечт / шагов — кнопки `.dreams-filter-btn`, активная `.dreams-filter-btn.active`.

| Состояние | Фон | Текст | Скругление |
|-----------|-----|-------|------------|
| Неактивный | `transparent` (по смыслу), лёгкая прозрачность | `#555` | `4px` |
| Hover | — | темнее | — |
| **Активный** | **`#4CAF50`** (как `var(--island-accent)`) | **`#fff`**, `font-weight: 600` | `4px` |

Классы и отступы см. в `index.html` в секции `.dreams-filter-btn` / `.dreams-filter-btn.active`.

**Применение:** любые горизонтальные «вкладки-чипы» в одном стиле с фильтром мечт (не подчёркивание как у ссылки, а заливка активного пункта).

### 1.1. Таблица мечт (макет колонок) — **контракт**

**Источник в коде:** `index.html` — таблицы `#dreams-table` и `#buddy-dreams-table`, класс `.dreams-table`, `<colgroup>` + классы колонок (`.dreams-col--*`). **Не** задавайте ширину «Мечте» фиксированной в пикселях; **запрещено** плодить альтернативные селекторы ширин для `td` без согласования с этим контрактом.

| Колонка | Смысл | Фиксированная ширина (база) | Примечание |
|--------|--------|----------------------------|------------|
| **№** | Порядковый номер | **`1.6em`** | ~половина прежних ~3.25em; 1–2 цифры, компактно. |
| **Шаги** | `выполнено/всего` (неудалённые), прогресс = фон ячейки | **`6.5ch`** (tabular-nums) | До `999/999` без подрезки; цифры одинаковой ширины. |
| **Мечта** | Текст мечты | **вся оставшаяся ширина** | `colgroup`: без `width` / `min-width: 0`, перенос слов; главная колонка продукта. |
| **Дедлайн** | Отображение, не шире дд.мм.гггг | **`10ch`** + гориз. отступы в `td` **6+6 px** (итого визуально = длина 10 цифр+точек) | `white-space: nowrap` в ячейке, шрифт как сейчас. |
| **Цена** | Сумма в руб. | **`13ch`** (tabular) | Максимальная **опорная** строка: `100 000 000` (11 цифр + 2 обычных пробела) — 13 позиций; сад на миллиард — отдельно. |
| **Статус** | 👁, **пикер** (`.status-picker-current` `min-width: 2em` + рамка), ✏, `gap: 8px` | **`8.5em`** | Три элемента в ряд; 5.5em давало **вылет** третьего; опора на фактическую вёрстку, не на «два значка». |

- **`table-layout: fixed`** + **`width: 100%`** — только **в паре** с `colgroup` по этой таблице, чтобы **Мечта** забирала **остаток** после фиксированных колонок, а не «съезжала» вся строка.  
- **Снимок готовой разметки/стилей** для бэкапа: **[Readme/stable/dreams_table.html](stable/dreams_table.html)** (см. [Readme/stable/README.md](stable/README.md)).  
- **Модалка шага/мечты** — **отдельные** селекторы (например `.modal-add-step`, `.modal-dream-table`); **не** смешивать с макетом таблицы мечт в ЛК.

## 2. Навигация кабинета «мечты | шаги | дневник | витрина»

**Классы:** `.cabinet-nav-link` на элементах `#header-link-dreams`, `#header-link-steps`, `#header-link-diary`, `#header-link-showcase`; активный пункт — `.cabinet-nav-link.active`.

**Логика активного пуна** (функция `updateCabinetNavActive` в `index.html`):

- **мечты** — активен при разделах `dreams` и `buddy`;
- **шаги** — при `steps` и `steps-buddy`;
- **дневник** — при разделе `diary` (события шагов из `dreams_steps_events`);
- **витрина** — при `showcase`;
- при `notifications`, `profile-edit` — ни один из четырёх основных не подсвечивается (это не основные вкладки кабинета).

**Визуальная стабильность хедера:**

- Строка имени: `.app-header-name-row` + имя `#profile-full_name` + группа `.app-header-name-actions` (**🔔**, **📚**) — без переноса между колокольчиком и библиотекой; при нехватке ширины имя обрезается многоточием (`text-overflow: ellipsis`), иконки остаются рядом.
- Пункты `мечты | шаги | дневник | витрина` должны помещаться в одну строку без переноса.
- Переключение `.active` не должно менять ширину текста (без «дёргания»): одинаковый `font-weight` и близкие метрики в пассивном/активном состоянии.
- Аватары в хедере (`.app-header-avatar`, `.app-header-buddy-slot`) всегда круги: фиксировать `width == height`, не тянуть через `align-self: stretch`.

### Возврат в рабочую область после действия (return-to-source)

**Принцип:** пользователь работает в некоторой **рабочей области** кабинета (раздел «мечты» / «шаги» / «бадди» / «витрина», режим списка или слайдера, раскрытый аккордеон и т.д.). После завершения действия, которое **инициирует перезагрузку данных** (сохранение, удаление, PATCH и последующий `refreshAfterDreamChange` и т.п.), интерфейс должен вернуть пользователя **в ту же рабочую область, из которой действие было вызвано**, а не в раздел, заданный устаревшим глобальным флагом от другого сценария.

**В коде (`index.html`):** в начале **`refreshAfterDreamChange`** вызывается **`captureCabinetReturnSection()`** — снимок видимого раздела по блокам `#diary-block`, `#steps-block`, `#buddy-block`, `#showcase-block`, `#notifications-block`, `#profile-edit-block`, иначе **`dreams`**. Значение кладётся в **`restoreSectionAfterStepSave`**, после `showProfile` / `loadBuddyDreams` выполняется **`switchMenu(section)`**. Дополнительно при необходимости восстанавливают мечту/прокрутку через **`dreamIdToRestoreAfterRefresh`** и связанную логику.

**Не делать:** полагаться только на значение, записанное при открытии одной конкретной модалки, если позже пользователь совершил другое действие в другом разделе без обновления этого флага.

## 3. Календарь даты (стандарт «Остров»)

**Эталон в коде:** один общий попап `#island-date-calendar-popover` и функции `uiDateToIso` / `isoToUiDate` / `initIslandDateCalendar` в `index.html`.

### Ввод и формат

| Правило | Значение |
|--------|----------|
| Поле ввода | `type="text"`, класс **`island-date-field`**, плейсхолдер **`дд.мм.гг`**, `maxlength="10"`, `inputmode="numeric"`, `autocomplete="off"` |
| Отображение и ввод | **дд.мм.гг** (день и месяц с ведущим нулём при нормализации; год — **2 или 4 цифры**, правила года см. `uiDateToIso`) |
| В API и БД | только **ISO** `YYYY-MM-DD`; перед отправкой — **`uiDateToIso`**, для показа из API — **`isoToUiDate`** |
| Ошибка ввода | класс **`island-date-field-invalid`** на поле (красная рамка) |

**Не использовать** нативный `type="date"` для полей ЛК в этом стандарте: формат и локаль зависят от ОС/браузера и не совпадают с дд.мм.гг.

### Поведение календаря

- **Открытие:** при **фокусе** на поле с `island-date-field` (отдельная кнопка «календарь» не нужна).
- **Позиция:** `position: fixed` под полем или над ним, если снизу не хватает места (`islandCalPosition`).
- **Закрытие:** `Escape` (сначала календарь, затем модалка — см. обработчик `keydown` в `index.html`), клик вне попапа и вне активного поля, **mousedown** с учётом ссылок-подсказок `.modal-deadline-hints` (клик по подсказке не закрывает календарь), `blur` поля с небольшой задержкой, если фокус не ушёл внутрь попапа.
- **Сетка:** неделя с **понедельника**; шапка — кнопки «‹ ›», выпадающие **месяц** (`#island-cal-month`) и **год** (`#island-cal-year`), сетка дней `#island-cal-grid`, классы ячеек **`island-cal-day`**, пустые — **`island-cal-day-empty`**, выбранный день — **`island-cal-day-selected`**.
- **Месяцы в селекте:** краткие подписи из массива **`ISLAND_CAL_MONTH_SHORT`** (Янв–Дек), полное название в атрибуте **`title`** у option.
- **Годы:** диапазон строится вокруг текущего просмотра и «сегодня» (см. `islandCalFillYearSelect`), глобальный коридор **1980–2120**.
- При вводе с клавиатуры, если дата распознана, месяц/год в попапе **синхронизируются** (`islandCalRefreshFromTyping`).

### Внешний вид (палитра)

- Контейнер попапа: класс **`island-date-calendar-popover`** — светлый фон с лёгким голубым градиентом, рамка **`#81d4fa`**, тень с синим оттенком (как в `index.html`).
- Селекты месяца/года: **`island-cal-month-select`** / **`island-cal-year-select`** — белый фон, рамка **`#64b5f6`**, текст **`#01579b`**, жирный шрифт.
- Активные элементы и акценты: тёмно-синие **`#01579b`**, выделение дня **`#002171`** (см. стили `.island-cal-day` в `index.html`).

### Где уже применяется

- Дедлайн мечты (создание / редактирование): поля `#new-dream-deadline`, `#edit-dream-deadline`.
- **Модалка шага — одна DOM (`#modal-add-step`), один эталон вёрстки, но динамический UI (progressive disclosure).** Базово показываются: `Название` и одно поле `Дата` (`#modal-add-step-date-from`). Дополнительные поля открываются кнопками `+` в одной строке под названием: время старта / окончания, повторяемость серии (`Дата окончания` + дни недели), цель по количеству (`#modal-add-step-target-wrap`). Для `edit-series` сохраняются кнопки области **«1 шаг / N шагов»**, а период серии показывается через `#modal-add-step-date-from-wrap`/`#modal-add-step-date-to-wrap`. Окно открывается из всех контуров через `openAddStepModal` / `openEditStepModal`; отдельную «вторую» модалку не создавать.
- Кнопки **«1 шаг»** / **«N шагов»** (`#modal-add-step-dates-scope-wrap`, `#modal-add-step-edit-scope-btns`): только в `edit-series`; в разметке **сразу под** строкой подсказки времени и **выше** ряда «Повторять» и блока дней недели, чтобы на узком экране они не оказывались «внизу» после длинного блока дней.
- Режим `edit-series` для **«N шагов»**: в расчёте и применении **сохранения** участвуют только шаги **без реакции** (не выполнены и не удалены) от даты текущего шага и дальше (`getSeriesMembersForEdit`). Состав серии для этого списка должен совпадать с компактной группой в мечте: при открытии модалки из компактной строки передаётся `_seriesSteps`, он же хранится на оверлее, пока модалка открыта, чтобы **не терять членов серии**, если раздел «Шаги» ещё не заполнял `stepsFlatCache`. Без `_seriesSteps` источник — шаги мечты из кэша / плоский список, как раньше.
- Поля **«Начало»** и **«Конец»** в `edit-series` показывают **фактический календарный диапазон серии**: минимальный и максимальный `deadline` среди всех **неудалённых** шагов этой серии в мечте (**включая уже выполненные**). Не подставлять «сегодня» вместо реального начала, не сужать период до только непройденных шагов и не подменять **«Начало»** датой открытого экземпляра (дата открытого шага — только в заголовке модалки «Редактировать шаг за …»). Для границ и предвыбора дней недели используется полный состав серии (`getSeriesMembersForCalendarBounds`); при открытии редактирования из **компактной строки серии** внутри мечты в модалку передаётся тот же набор шагов, что в схлопнутой группе, чтобы поведение совпадало с разделом «Шаги».
- Плейсхолдеры **«Время с»** / **«Время по»** в модалке шага: **`00:00`**.
- В `edit-series` поля периода и времени остаются выровненными по высоте (`38px`), а в `create`/`edit-single` видимость этих полей управляется state-driven раскрытием.
- Модалка финансовых шагов: `#modal-fin-end-date`.
- Перенос даты шага: `#step-move-deadline` в `#modal-step-move` (§6).

Любое новое поле даты в кабинете — **тот же маркер `island-date-field`** и тот же попап (после вызова `initIslandDateCalendar` поля, уже в DOM при загрузке, привязываются автоматически; для полей, добавляемых динамически позже, нужно либо вызывать ту же логику привязки, либо держать поля в статическом HTML до `initIslandDateCalendar`).

## 4. Как добавлять новые стандарты

1. Описать паттерн здесь (таблица или короткий список свойств).
2. Вынести общие правила в **один** набор классов в `index.html` (или позже — в общий CSS, если вынесем стили).
3. В `Readme/CHANGELOG.md` — одна строка со ссылкой на этот файл (и при необходимости кратко — что именно добавлено в стандарты).

## 5. Обязательная синхронизация окон со стандартом

- Окна **создания** и **редактирования** (включая контур бадди) строятся из одного стандарта и общей DOM/JS-логики.
- Любое изменение одного окна (layout, кнопки, подписи, поведение) обязательно влечёт:
  1) обновление стандарта в этом файле;  
  2) обновление всех окон, привязанных к этому стандарту.
- Запрещено оставлять «частный» UI-фикс только в одном контуре (`dreams` или `buddy`) без зеркалирования в другом, если это один и тот же паттерн.

**Связанные документы:** вёрстка кабинета — [Readme/structure.md](structure.md).

## 6. Шаги: единая строка (эталон temp), неотмеченные после дня дедлайна, модалки, дневник

**Где в коде:** разметка и стили — классы `.step-unified-inner`, `.step-actions-left`, `.step-actions-right`, `.step-main-col`, `.step-time-under`, кнопки `.step-done-btn`, `.step-waived-btn`, `.step-waived-clear`, `.step-comment-btn`, `.step-move-btn`, `.step-edit`, `.step-delete`; сборка — `buildUnifiedStepRowHtml`, группировка по дням — `buildDreamStepsListGroupedHtml`, обработчики — `wireUnifiedStepRow` / `wireDreamStepsList` в `index.html`.

**Календарь и граница дня:** у шага дедлайн — одна дата (`YYYY-MM-DD`). **Время суток шага** (`start_time` / `end_time`) **не** влияет на то, «пора ли требовать отметку» и на `completed_late`: важна только **смена календарного дня** в **локальном времени браузера** (полночь после дня `deadline` → с этого момента шаг без ✓/− считается **неотмеченным после дня**; в API на бэкенде для `completed_late` при `PATCH` клиент передаёт `completed_late` с той же логикой, иначе сервер использует эвристику «00:00 UTC следующего дня после даты deadline»).

### Раскладка строки и группировка

- **Десктоп (>=769px):** две горизонтальные панели в одной строке шага — слева 3 действия (**✓ − 💬**), справа 3 действия (**⇢ ✏️ 🗑️**). Позиции фиксированные: слева и справа от основной колонки, без вертикального стекования.
- **Мобильный (<=768px):** вместо двух панелей показывается кнопка **`⋮`**. По клику открывается выпадающее меню действий шага с иконкой и подписью: **✓ Выполнил вовремя** (текст кнопки действия), **− Не выполнил**, **💬 Прокомментировать**, **⇢ Перенести**, **✏️ Редактировать**, **🗑️ Удалить** (набор пунктов может сокращаться для недоступных состояний/исключений).
- **Слева (логическая группа действий):** **✓** (выполнено), **−** (не выполнен / waived), **💬** (комментарий в дневник). Не нативный чекбокс: **✓** — кнопка с состоянием «включено» (класс `step-done-on`), та же логика, что была у чекбокса (в т.ч. финансовый шаг — отдельное подтверждение суммы, без новой «идеальной» логики).
- **По центру:** время и название в **одной строке** (время слева перед названием), далее бейджи/интервал/количество. Дублирующей второй строки времени не должно быть.
- **Справа:** **⇢** (перенос дедлайна), **✏️** (редактирование, существующая модалка шага), **🗑️** (мягкое удаление `deleted`, иконка корзина).
- **Заголовки групп:** при сортировке по дедлайну перед первым шагом каждого нового дня — строка-разделитель (в списке «Шаги» — `li.step-day-divider`, в аккордеонах/карточках — `div.step-day-divider`) с датой дня в формате **дд.мм.** (как `formatStepsListDayDivider`).

**Кнопки в строке:** у всех — сброс глобального `button { width: 100% }`: `width: auto`, `flex: 0 0 auto`, разумные min/max (см. стили `.step-done-btn`, `.step-waived-btn`, … в `index.html`).

### Неотмеченные после дня дедлайна (фильтр «Проср.»)

- Одинаково для шагов **с** временем и **без**: **с полуночи первого календарного дня после** `deadline` (локальное время браузера) шаг считается **ожидающим отметки**, пока не стоит «выполнено» или «не выполнен» (но **не** `deleted`).
- Фильтр кнопки **«Проср.»** показывает такие шаги; подпись **«Проср.»** в сегменте фильтра **красная** (только текст), пока есть хотя бы один такой шаг; после выставления статусов по всем — снова нейтральный серый.
- Отметка **✓**, если день дедлайна уже прошёл, ведёт к `#modal-step-done-late` (опциональный комментарий); в БД выставляется **`completed_late`** = true (подпись в UI: **«Выполнил с опозданием»**). **Не** смешивать с бейджем «не отмечен» (шаг ещё **не** закрыт).

### Статусы (смысл)

| Понятие | Поле | Смысл |
|--------|------|--------|
| Выполнил вовремя | `completed`, `completed_late = false` | Отметка «выполнено» **до** полуночи после дня `deadline` (локальный календарь). |
| Выполнил с опозданием | `completed`, `completed_late = true` | Отметка «выполнено» **после** этой полуночи. |
| Не выполнен («минус») | `waived = true`, `completed = false` | Фиксация «не сделал» для статистики; **не** удаление. |
| Удалён | `deleted = true` | Корзина, «как не создавал». |

### Модалки и дневник (без пустых событий)

- **`#modal-step-waived`:** **«Пропустить»** — `waived: true` без текста; **«Сохранить»** — только при непустом поле (`waived` + `note`); **«Отмена»** / клик по оверлею / `Escape` — без изменений. Слово **«обязательно»** в UI не использовать.
- **`#modal-step-comment`:** только комментарий (`PATCH` с `note`); **«Сохранить»** — при непустом тексте; **«Пропустить»** / **«Отмена»** — закрыть без записи в `dreams_steps_events` и без пустых полей в UI.
- **`#modal-step-move`:** поле даты — стандарт «Остров» (`#step-move-deadline`, класс `island-date-field`, календарь §3). **«Пропустить»** — только новый `deadline` (без `note`). **«Сохранить»** — тоже при валидной дате; если комментарий введён — добавить `note` в тот же `PATCH`; пустой комментарий не отправлять. Пустые события дневника не создавать.
- После успешных операций из модалок и из строки (где нужно обновить кэш мечт) вызывается **`refreshAfterDreamChange`** с при необходимости **`dreamIdToRestoreAfterRefresh`**, чтобы списки мечт и шагов совпадали с БД после перезагрузки страницы.

**Таблица мечт (десктоп):** полный контракт ширин и ведения **Мечты** — **§ 1.1** выше. **Шаги:** прогресс = **левый залив** фона `td` (без вложенных «рельс»); **№** — только цифра; **Мечта** — вся оставшаяся ширина.

Данные дневника — `dreams_steps_events`; список — `GET /steps/events?user_id=…` (владелец мечт; в режиме **бадди** — `user_id=владелец` и `viewer_id=я`, как у `GET /dreams`). Ориентир строки: **дата — шаг — текст — статус**. Типы событий см. [Readme/tables.md](tables.md) (`event_type`).

### Где применяется и как называется режим

- **«Расписание» (детально):** раздел **«Шаги»** (`steps-list`) — полный поэкземплярный список с датами/временем и всеми действиями на конкретном шаге.
- **«Серии шагов» (компактно):** внутри конкретной мечты (аккордеоны десктоп + мобильные карточки/список, включая бадди) — повторяющиеся экземпляры схлопываются в одну строку серии: название, знак качества, количество, интервал дат (`01.04 по 21.04`), время.
- **Счётчик серии в названии шага (`N/M`):** `N` — **исторический порядковый номер** шага в серии (из `series_index`; если поля нет — fallback по сортировке дат), `M` — **фактическое число неудалённых шагов** серии на текущий момент. Счётчик **не** считается «из оставшихся» и должен быть одинаковым во всех контурах (раздел «Шаги», компактная серия внутри мечты, бадди, модалки). Перенос шага на другую дату **не должен** менять его `N`.
- Список удалённых шагов в разделе «Шаги» остаётся по шагам (действие — восстановление).

**Вёрстка компактной серии:** левая колонка действий скрыта (`step-compact-series-inner`); в мобильном меню `⋮` используется тот же единый набор действий, что и в других контурах (**✓ − 💬 ⇢ ✏️ ✕**), при недоступности отдельные пункты показываются disabled с `title`-подсказкой причины. Основной текст (время + название + бейджи + интервал) прижат к левому краю без пустых спейсеров.

Для компактной серии **✓** и **−** (и пункты меню `⋮` с тем же смыслом) действуют **только на неотмеченные после дня дедлайна** экземпляры серии (та же логика, что в §«Неотмеченные…»): кнопки и соответствующие пункты меню **скрываются**, если таких экземпляров нет. В строке при их наличии — бейдж **«не отмечено N»**; подсказки и текст `#modal-step-series-confirm` — с формулировками **«неотмеченные шаги»**.

Для уже выполненного шага/серии в контуре **внутри мечты**: кнопка **✓** серого вида **disabled** (не кликабельна), повторное действие «выполнить» не запускается; подсказка **«Шаг уже выполнен»** / **«Серия шагов уже выполнена»** — только при наведении (`title`), без браузерных `alert`. В строке шага в этом же контуре добавляется зелёный бейдж **«выполнено»** (рядом с при необходимости бейджами «не отмечен», «не выполнен», «Выполнил с опозданием»).

**Убрать шаги из расписания:** кнопка **✕** открывает `#modal-step-series-delete` (заголовок «Удаление шагов серии»; варианты **Все** / **Прошлые** / **Будущие** — с классом `island-btn-outline`, **Отмена** — `secondary`, см. §0.1). Смысл дат: по календарной дате дедлайна в **UTC** — *Прошлые* (дата строго раньше сегодня); *Будущие* — сегодня и позже; *Все* — вся серия. Недоступные варианты disabled. Редактирование — стандартная модалка шага.

### Исключения (до отдельной задачи)

- **Финансовая цель в аккордеоне мечты:** таблица месяцев с планом/фактом и полями ввода — **не** полная панель ✓ − 💬 / ⇢ (перенос даты месяца там не используется). В общем списке «Шаги» для синтетических фин-строк — текущая логика: **✓** открывает подтверждение суммы; **− / 💬 / ⇢** скрыты; справа остаются **✏️** и **🗑️** (если доступно управление).
- **Книги** (мечти с правилом чтения): без изменения текущих веток; если позже появится отдельное поле «комментарий у шага» — пустое не заполнять искусственно.
