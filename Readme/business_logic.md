# Бизнес-логика (фрагменты)

## Дневник шагов (`dreams_steps_events`)

**Дневник задуман как журнал рефлексии («слабых мест»):** в ленту попадают только записи с **непустым пользовательским текстом** после trim (`note` в теле PATCH или эквивалент в событии).

- В дневнике **не хранятся «успешные» события** (`completed`, `series_completed`) — ни новые, ни исторические; они удаляются из `dreams_steps_events`.
- Отметка шага без текста (в том числе «выполнено», «пропустить комментарий» при опоздании, перенос дедлайна без комментария, waived без текста) **обновляет статус шага в БД**, но **не создаёт** строку дневника.
- События «успеха» отслеживаются интерфейсом «Шаги» (в т.ч. фильтр «Готово»), а дневник остаётся каналом для причин/рефлексии и явного действия «Комментарий».

Подробнее про UI модалки опоздания и переключатель «не спрашивать» — [UI-standards.md](UI-standards.md) §6.

## Библиотека: привязка книги к шагу чтения

- Для книги (`dream_books`) можно задать желаемую дату прочтения (`deadline`); просрочка для статусов кроме `finished` подсвечивается в UI.
- Статусы книги: `planned`, `reading`, `listening`, `finished` (в интерфейсе: «Планирую», «Читаю», «Слушаю», «Прочитано»).
- Связь книги и шага чтения хранится явно в `dream_books.linked_step_id`. Несколько книг могут ссылаться на **один и тот же** шаг (одна итерация); ограничение «одна книга — один шаг» на уровне БД не задаётся.
- В UI и API кандидаты для привязки **схлопываются по серии** (`series_id` или базовое имя шага + время): в выпадающем списке одно наименование логического шага; в базу пишется id **итерации с дедлайном на локальный «сегодня»** клиента (`local_date` в запросе кандидатов). Подпись книги к шагу в расписании сопоставляется по той же серии; у строки с дедлайном **якорного дня** (сегодня в списке шагов или день отчёта) для серии показывается суффикс с книгой(ами). Выбор шага и **Связать / Отвязать** — в той же модалке, что и редактирование книги (без отдельного окна). В списке библиотеки подписи **«Шаг: …»** и **«Мечта: …»** (вторая строка) строятся из кэша расписания или кэша мечт; в модалке для привязки показывается только название шага; габариты модалки — по [UI-standards.md](UI-standards.md) §2.
- При переходе книги в `finished` связь очищается автоматически (`linked_step_id = NULL`).
- Кандидаты шагов для привязки ищутся по keyword-match в заголовке шага (`читать`, `чтение`, `книга`, `аудиокнига`, `слушать`) без ИИ; список ключей расширяем по мере наблюдений.
- Если к одному шагу привязано несколько активных книг (`reading`/`listening`), в UI показывается агрегированная подпись вида «читаю одну из N книг (...)».
- Автосоздание отдельной мечты/шага при отсутствии шагов чтения — отложено как следующий этап.
